Your central view for monitoring incidents, investigating root causes, and managing your services.

## Incidents

The incidents page is where you spend most of your time.

### Incident list

- **Filter** by status (Escalated, Suppressed, Resolved), service, or time range
- **Search** by title or description
- **Sort** by creation time, severity, or status

### Incident detail

Click any incident to see its details across four tabs:

<Tabs>
  <Tab title="Overview">
    | Field | Description |
    |-------|-------------|
    | **Status** | RECEIVED, INVESTIGATING, ESCALATED, SUPPRESSED, RESOLVED |
    | **Confidence** | How confident Wraith is in its verdict (0–100%) |
    | **Service** | The affected service |
    | **Alert Source** | Which monitoring tool triggered the alert |
    | **Timeline** | Step-by-step pipeline progress |
    | **Summary** | AI-generated root cause and impact description |
  </Tab>
  <Tab title="Evidence">
    - **Root Cause Signals** — code paths, queries, or calls identified as the failure point
    - **Dependency Graph** — services involved in the incident
    - **Verification Report** — confidence level and supporting evidence
  </Tab>
  <Tab title="Logs">
    - **Pipeline Logs** — logs from each analysis step
    - **Application Logs** — relevant logs from around the alert time
  </Tab>
  <Tab title="AutoFix">
    When AutoFix runs:
    - **Fix Status** — attempted, succeeded, or failed
    - **PR Link** — direct link to the GitHub pull request
    - **Patch Diff** — the generated code change
  </Tab>
</Tabs>

### Actions

| Action | Description |
|--------|-------------|
| **Resolve** | Mark the incident as fixed |
| **Suppress** | Mark as noise or false positive |
| **Feedback** | Rate the verdict accuracy — improves future triage |

## Services

Each service shows its name, health status (`up`, `degraded`, `down`, `unknown`), when it was last seen, and recent incident count.

The **dependency graph** shows service-to-service relationships. Use it to understand blast radius when a service fails.

## Runbooks

Structured incident response procedures.

<Steps>
  <Step title="Create">
    Click **Create Runbook** — define title, service, trigger conditions, and ordered steps.
  </Step>
  <Step title="Match">
    When an incident matches a runbook's trigger, Wraith suggests it.
  </Step>
  <Step title="Execute">
    Work through the steps, marking each as done. History is recorded for retrospectives.
  </Step>
</Steps>
